### Rapport Technique : Implémentation Optimisée de la FSM BACnet/MSTP sur ESP32-S3 \#\#\#\# 1\. Analyse de la Couche de Liaison de Données MS/TP Le protocole BACnet (ASHRAE 135\) repose sur une "Collapsed Architecture" (architecture réduite) par rapport au modèle OSI. Cette structure fusionne les couches Session, Transport et Présentation directement dans la couche Application pour optimiser l'efficacité des communications en environnement contraint. Dans ce cadre, la couche de liaison **MS/TP (Master-Slave/Token-Passing)** est fondamentale : elle permet un passage de jeton robuste à faible coût sur une topologie RS-485. Pour garantir l'interopérabilité des systèmes de gestion technique du bâtiment (GTB) — incluant le CVC, l'éclairage et la sécurité — l'implémentation doit respecter strictement les contraintes temporelles de la norme. Sur un microcontrôleur haute performance comme l'ESP32-S3, la précision des fenêtres de silence et de réponse est le facteur critique de stabilité du réseau : \* **Tframe\_gap (20 bit times) :** Temps maximum entre deux octets d'une même trame (\~520 µs à 38400 bps). \* **Tusage\_timeout (20 ms à 100 ms) :** Délai d'attente pour qu'un nœud commence à utiliser le jeton. \* **Treply\_timeout (255 ms max) :** Temps alloué pour recevoir le premier octet d'une réponse. \#\#\#\# 2\. Architecture Matérielle et Gestion de l'UART Non-Bloquante Le protocole MS/TP opérant en half-duplex sur RS-485, la gestion du signal RTS (Request To Send) pour le contrôle du driver de ligne est critique. L'ESP32-S3 permet une gestion matérielle native de ce mode, minimisant ainsi la latence de commutation directionnelle. \#\#\#\#\# Configuration du Driver UART RS-485 Nous configurons l'UART pour traiter les interruptions via une file d'événements et forçons le mode UART\_MODE\_RS485\_HALF\_DUPLEX. Le seuil de la FIFO est abaissé pour satisfaire Tframe\_gap. cpp \#include \#include \#include \#include  class MstpUartHandler { public: static constexpr int UART\_PORT \= UART\_NUM\_1; static constexpr int BUF\_SIZE \= 1024; void init() { uart\_config\_t uart\_cfg \= { .baud\_rate \= 38400, .data\_bits \= UART\_DATA\_8\_BITS, .parity \= UART\_PARITY\_DISABLE, .stop\_bits \= UART\_STOP\_BITS\_1, .flow\_ctrl \= UART\_HW\_FLOWCTRL\_DISABLE, .rx\_flow\_ctrl\_thresh \= 122, .source\_clk \= UART\_SCLK\_DEFAULT, }; ESP\_ERROR\_CHECK(uart\_param\_config(UART\_PORT, \&uart\_cfg)); // Configuration RS-485 : gestion automatique du RTS ESP\_ERROR\_CHECK(uart\_set\_pin(UART\_PORT, 17, 18, 19, UART\_PIN\_NO\_CHANGE)); ESP\_ERROR\_CHECK(uart\_driver\_install(UART\_PORT, BUF\_SIZE, BUF\_SIZE, 20, \&m\_uart\_queue, 0)); ESP\_ERROR\_CHECK(uart\_set\_mode(UART\_PORT, UART\_MODE\_RS485\_HALF\_DUPLEX)); // Optimisation FIFO pour Tframe\_gap ESP\_ERROR\_CHECK(uart\_set\_rx\_timeout(UART\_PORT, 2)); // \~2 symboles } private: QueueHandle\_t m\_uart\_queue; };  \#\#\#\# 3\. Conception de la Machine à États Finis (FSM) Master Node La classe MstpMasterFSM encapsule l'intelligence du protocole. Elle gère les variables normatives : TS (This Station), NS (Next Station) et PS (Poll Station). \#\#\#\#\# Synthèse des 9 États ASHRAE 135 | État | Rôle | Transition Critique | | \------ | \------ | \------ | | **INITIALIZE** | Setup des variables TS et NS. | Vers IDLE après Tno\_token. | | **IDLE** | Écoute passive du média. | Vers RECEIVE\_PA\_TOKEN si préambule (0x55 0xFF) reçu. | | **WAIT\_FOR\_TOKEN** | Attente d'un jeton spécifique à TS. | Vers USE\_TOKEN si Destination Address \== TS. | | **USE\_TOKEN** | Transmission de données ou Poll For Master. | Vers PASS\_TOKEN ou WAIT\_FOR\_REPLY. | | **WAIT\_FOR\_REPLY** | Attente de la réponse du destinataire. | Vers IDLE après Treply\_timeout ou réception Data. | | **PASS\_TOKEN** | Envoi du jeton à NS. | Vers IDLE après confirmation d'activité. | | **NO\_TOKEN** | Détection d'un réseau inactif. | Vers INITIALIZE pour générer un nouveau jeton. | | **ANSWER\_GROUP\_PDU** | Traitement des broadcasts. | Retour immédiat vers IDLE. | | **PROCEED\_TO\_NEXT** | Incrémentation de la station cible (NS). | Vers PASS\_TOKEN. | \#\#\#\# 4\. Optimisation pour l'ESP32-S3 et FreeRTOS Pour garantir le déterminisme requis par la norme ASHRAE 135, nous isolons la FSM sur le second cœur du processeur. \* **Partitionnement Multi-Cœur :** \* **Core 0 :** Gestion des interruptions UART de bas niveau et remplissage d'un buffer circulaire thread-safe. \* **Core 1 :** Exécution de la tâche MstpMasterTask avec une priorité configMAX\_PRIORITIES \- 1\. \* **Gestion Mémoire :** Utilisation exclusive de std::array et de variables membres statiques. L'absence d'allocation dynamique (new/malloc) dans la boucle step() prévient toute latence induite par le Garbage Collector du tas. \* **Synchronisation :** Utilisation de std::atomic pour signaler la présence de nouvelles trames entre les cœurs sans blocage de mutex. \#\#\#\# 6\. Validation et Diagnostic Le respect des timings est validé par un moniteur de gigue (jitter) intégré : 1\. **Mesure Delta :** Utilisation de esp\_timer\_get\_time() pour capturer l'intervalle entre l'arrêt de la transmission (UART\_TX\_DONE) et le début de la réception du premier octet de réponse. 2\. **Logging Allégé :** Les transitions d'état sont enregistrées dans un buffer circulaire en RAM (MstpHistory). En cas d'erreur de protocole, ce buffer est déversé via ESP\_LOGI uniquement lors des phases d'IDLE prolongées, évitant ainsi d'impacter le bus RS-485 par des délais d'affichage. \#\#\#\# 7\. Conclusion et Recommandations Cette architecture optimisée assure une conformité rigoureuse aux spécifications MS/TP tout en tirant parti de la puissance de calcul de l'ESP32-S3. L'isolation de la FSM garantit que les délais Tframe\_gap et Treply\_timeout sont respectés avec une précision sub-milliseconde. **Étapes d'intégration supérieures :** \* **Modèle d'Objets :** Mapper les registres matériels sur les **23 types d'objets standards** (Analog Input pour les capteurs, Binary Output pour les actionneurs). \* **Services Applicatifs :** Implémenter les BIBBs (BACnet Interoperability Building Blocks) pour le partage de données, notamment les services ReadProperty et WriteProperty décrits dans le standard. \* **Interopérabilité :** Cette couche logicielle constitue la fondation nécessaire pour une certification BTL (BACnet Testing Laboratory), assurant une intégration transparente avec les superviseurs du marché.
